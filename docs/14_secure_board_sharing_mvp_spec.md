# Secure Board Sharing MVP Spec

## Goal

Build a first sharing system for Paapan that feels simple like Miro / FigJam, but keeps the security model strict from day one.

For MVP, sharing should mean:

- One board can stay `private` or be shared via `link view only`
- Owner can copy link, turn sharing off, and regenerate link
- Shared board can be opened without editor access
- Shared board can optionally be duplicated into another user's workspace

This spec intentionally avoids public edit links for the first version.

## Product Model

Use this mental model:

- `Workspace` in the current codebase is functionally the same as a `board`
- Sharing happens per board, not per account
- Access defaults to private
- Public access is always token-based, never by raw board id

This matches the strongest pattern across Miro, Figma/FigJam, and tldraw-based apps:

- one shareable document / board
- private by default
- explicit permission change by the owner

## Security Principles

1. Private by default
2. Share uses random opaque tokens, not internal ids
3. Public access only gets the minimum board payload needed to render
4. Owner actions always require authenticated ownership checks
5. Private image assets must not remain globally public
6. Regenerating a link must immediately invalidate the old link
7. Public endpoints should return `404` for both missing and unauthorized resources to avoid resource discovery
8. Public endpoints should be rate-limited

## MVP Permission Model

For MVP, keep permissions intentionally small:

- `private`
- `link_view`

Additional flags:

- `allow_duplicate`: `true | false`

Not in MVP:

- `comment`
- `edit via link`
- `invite by email`
- shared team boards

## Data Model Changes

Add board sharing fields to the `workspaces` table.

Recommended columns:

```sql
alter table public.workspaces
add column share_visibility text not null default 'private'
check (share_visibility in ('private', 'link_view')),
add column share_token_nonce text unique,
add column allow_public_duplicate boolean not null default true,
add column shared_at timestamptz,
add column share_updated_at timestamptz;
```

Optional but recommended later:

```sql
create table public.workspace_share_access_events (
  id uuid primary key default gen_random_uuid(),
  workspace_id uuid not null references public.workspaces(id) on delete cascade,
  access_type text not null check (access_type in ('view', 'duplicate')),
  token_prefix text,
  created_at timestamptz not null default now(),
  ip_hash text,
  user_agent text
);
```

### Why use nonce + signature instead of storing a full public token

Do not store a reusable share token string in the database.

Flow:

- generate a random `nonce` server-side
- store only that `nonce`
- build the public token as `nonce.signature`
- derive the signature from `workspaceId + nonce + server secret`
- verify the signature on read

That way:

- the database still does not contain the final reusable public token
- links can be regenerated by replacing the nonce
- token validation stays server-authoritative

## URL Design

Use a public route like:

```text
/b/[shareToken]
```

Avoid using:

- `/workspace/[id]`
- `/share/[workspaceId]`

Reason:

- internal ids should stay internal
- token route makes revocation simple

## API Design

### Owner APIs

Authenticated endpoints:

- `POST /api/boards/:id/share/enable`
- `POST /api/boards/:id/share/disable`
- `POST /api/boards/:id/share/regenerate`
- `PATCH /api/boards/:id/share/settings`

Rules:

- require auth
- require board ownership
- do not trust client-provided ownership

Payload example:

```json
{
  "visibility": "link_view",
  "allowDuplicate": true
}
```

### Public read API

Public endpoint:

- `GET /api/public/board/[shareToken]`

Behavior:

- parse the token into `nonce` and `signature`
- look up the board by `share_token_nonce`
- verify the signature server-side
- confirm `share_visibility = 'link_view'`
- return `404` if missing or disabled
- return sanitized board payload only

### Duplicate API

Endpoint:

- `POST /api/public/board/[shareToken]/duplicate`

Rules:

- only works if `allow_public_duplicate = true`
- for MVP, require login
- creates a new private board owned by the duplicating user
- strips the original sharing metadata from the new copy

## Sanitized Public Payload

Public response must include only what is necessary to render the board:

- board name
- nodes
- edges
- frames
- strokes
- arrows
- updated timestamp

Do not include:

- owner email
- user id
- admin flags
- subscription metadata
- credit data
- raw storage internals that are not needed by the client

Suggested shape:

```ts
type PublicBoardPayload = {
  name: string
  nodes: CanvasNodeType[]
  edges: Edge[]
  frames: FrameRegion[]
  strokes: DrawingStroke[]
  arrows: ArrowShape[]
  updatedAt: string
  allowDuplicate: boolean
}
```

## Image Security Plan

This is the most important architectural decision for secure sharing.

### Current state

Image nodes already persist `storagePath`:

- [types/index.ts](</C:/Users/Yosia/Downloads/Work/project table/spatial-ai-workspace/types/index.ts:126>)

This is good. It means we can migrate without redesigning the image node schema.

### Required direction

Make image storage private and treat `storagePath` as the canonical reference.

New rules:

- persisted node stores `storagePath`
- runtime rendering uses a short-lived resolved URL
- `src` becomes a session/runtime field, not the security boundary

### Migration approach

1. Stop creating public buckets for new uploads
2. Use a private bucket
3. Persist `storagePath` as the durable identifier
4. Resolve signed URLs when:
   - owner loads their own board
   - public shared board is materialized server-side
5. Gradually migrate existing public assets if needed

### Resolution strategies

Best MVP choice:

- public board API rewrites image nodes with short-lived signed URLs before returning the payload

This keeps the client simple and avoids exposing permanent asset URLs.

## Public Rendering Mode

Shared board page should open in a strict read-only canvas mode.

Disable:

- node editing
- AI actions
- upload
- delete
- toolbar creation actions
- settings that mutate board content

Allow:

- zoom
- pan
- open links
- export image if desired later
- duplicate if enabled

## Rate Limiting

Public endpoints should have lightweight rate limiting:

- per IP
- per token
- stricter on duplicate endpoint than view endpoint

Suggested MVP:

- public view: generous
- duplicate: tighter
- regenerate: owner-only and authenticated

## Regenerate Link Rules

When the owner regenerates a link:

- create a new token
- replace `share_token_nonce`
- update `share_updated_at`
- old link must stop working immediately

No grace period in MVP.

## Logging And Observability

For MVP, keep it light:

- `shared_at`
- `share_updated_at`
- optional access event table later

Nice next step:

- count total link opens
- count duplicates from shared boards

## Suggested Implementation Order

### Phase 1: secure foundation

1. Add sharing fields to `workspaces`
2. Add token generation and signature helper
3. Add owner share settings API
4. Add public board read API

### Phase 2: secure asset delivery

5. Move image delivery toward signed URLs
6. Add private-aware board materialization for image nodes

### Phase 3: UI

7. Replace current export-only share panel with sharing modal
8. Add public read-only board page
9. Add duplicate action

## Non-Goals For MVP

Do not build these yet:

- public edit links
- anonymous commenting
- multi-user live collaboration on shared public boards
- workspace-level sharing
- team permissions
- version history

## Recommendation

If we want Paapan to feel mature and safe, the first shipping version should be:

- private by default
- link view-only
- allow duplicate
- regenerate link
- owner-only sharing controls
- private image strategy started immediately

This is the smallest version that still respects real security boundaries.
